Una capa operativa local para trabajar con agentes IA en Pull&Bear.

El proyecto centraliza contexto, MCPs, skills y utilidades para convertir preguntas técnicas o funcionales en análisis accionables, tickets bien formados, diagnósticos y soporte operativo.

Propósito principal
Menos fricción. Más contexto. Mejor salida funcional.

No es una aplicación de producto final: es un sandbox de trabajo para agentes, análisis Pull&Bear, pruebas de herramientas, consultas Snowflake y soporte ecommerce.

Mapa de valor

Cuatro capas que convierten contexto disperso en decisiones accionables.

1. MCPs

Puentes hacia sistemas reales: Jira, GitHub, Snowflake, Zenit, SRE, Figma, Miro, documentación, GrowthBook, navegador y auditorías web.

2. Skills

Procedimientos reutilizables que reducen improvisación: análisis, creación Jira, diseño API, SDUI, observabilidad, CI/CD, Sonar, comunicación y reporting.

3. Utilidades locales

Scripts, prompts, templates, servidor Snowflake MCP y extensión CTR Review para resolver casos Pull&Bear con menos pasos manuales.

4. Reglas de rol

El agente trabaja como analista funcional senior: no inventa, traduce lo técnico a valor, y no crea Jira sin confirmación explícita.

Utilidades locales

Además de conectar herramientas, el sandbox incluye piezas locales para resolver casos concretos.

CTR Review

Extensión Chrome y backend local para mostrar CTR de productos sobre parrillas Pull&Bear Web usando vistas Zenit en Snowflake.

  • Detecta productos visibles.
  • Consulta impresiones y clicks.
  • Pinta overlays de CTR en la parrilla.

Snowflake MCP

Servidor local Python para consultas Snowflake con autenticación SSO externalbrowser y configuración Pull&Bear.

  • Consultas de datos.
  • Muestras de tablas.
  • Análisis de estructura y clustering.

AI Context

Prompts, scripts y templates importados desde el ecosistema ecommerce para acelerar tareas recurrentes.

  • Refresco de token ITX.
  • Monitor de major incidents.
  • Templates de testing Zenit.

Guardrails

El proyecto también define límites para proteger calidad y confianza.

No inventar

Si falta información obligatoria, se pregunta antes de actuar. Si no hay evidencia, se explicita la limitación.

No crear Jira sin permiso

La creación de issues exige resumen previo y confirmación explícita del usuario.

No tratar el sandbox como producto real

El workspace local es un entorno de análisis y pruebas salvo indicación expresa.
