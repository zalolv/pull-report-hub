Informe funcional: llamadas residuales a rutas /category

Revisión de Web, iOS y Android para entender en qué journeys de negocio podrían seguir generándose llamadas legacy tras la migración. El foco está en uso funcional real, no en falsos positivos de analítica, navegación o constantes.

Pull&Bear Repos revisados por GitHub CLI Rama principal revisada: develop Fecha: 2026-06-22

1. Resumen ejecutivo

La investigación apunta a que las llamadas residuales no parecen venir de un único punto común, sino de varios journeys históricos ligados a catálogo: carga de árbol de categorías, apertura de PLP, detalle de producto desde categoría, stock de categoría, deeplinks y preview CMS.

Riesgo alto

Android

Conserva endpoints Retrofit reales con rutas /category y rutas derivadas /category/{categoryId}/product, /detail y /stock. Es la plataforma con mayor probabilidad de explicar tráfico residual.

Riesgo medio

iOS

No se ve una URL literal /category, pero existe fallback legacy mediante client?.categories si la flag de StoreFront Menu está desactivada.

Riesgo medio/bajo

Web

No se identifica llamada directa literal a /category en el repo. Sí hay acciones Corefront relacionadas con categoría cuyo mapeo final vive fuera del repositorio.

Lectura de negocio: si el tráfico residual aparece en observabilidad como peticiones a /category, el primer sospechoso es Android, especialmente en flujos de menú/categoría, PLP, PDP desde categoría, stock y deeplinks. iOS dependería de configuración. Web requiere validar el mapeo externo de acciones.

3. Android: principal foco de llamadas residuales

Android mantiene definiciones de red reales con rutas legacy de categoría. A nivel negocio, no son solo llamadas técnicas aisladas: están conectadas con journeys activos o históricamente activos de catálogo.

Flujo Endpoint observado Interpretación funcional Referencias
Árbol de categorías / menú GET catalog/store/{storeId}/{catalogId}/category Carga la jerarquía de categorías para construir navegación y secciones. CategoryWs.kt, GetWsCategoryListUC.java, CategoryRepository, CategoryListPresenter
Detalle o validación de categoría GET .../category/{categoryId} Se usa al navegar a una categoría, validar deeplinks, resolver redirecciones o recuperar categoría si no está en caché. GetWsCategoryUC.java, CategoryViewModel, DeepLinkManager
PLP / productos de categoría GET .../category/{categoryId}/product Obtiene productos, filtros o datos iniciales para pintar la parrilla de categoría. ProductWs.kt, ProductGridApi.kt, GetWsProductListUC.java, GridViewModel
PDP desde PLP GET .../category/{categoryId}/product/{productId}/detail Carga detalle de producto manteniendo contexto de categoría. ProductAdapterApi.kt, GetWsProductDetailUC.java, GetLegacyProductDetailUseCaseImpl
Stock de categoría GET .../category/{categoryId}/stock Calcula disponibilidad asociada a los productos de una categoría. StockWs.kt, GetWsCategoryStockListUC.java, GetWsProductListUC.java
SEO/deeplink GET .../mapper/category/seo-id/{seoId} Resuelve una categoría desde identificador SEO. No es la colección /category, pero sí mantiene dependencia funcional con categoría. GetWsCategoryIdFromSeoIdUC.kt, CategoryWs.kt
CMS preview Variantes de /category y /category/{categoryId}/product con parámetros de idioma/catálogo Puede explicar tráfico residual interno o de validación de contenidos, no necesariamente de cliente final. getCategoryListCMSPreview, getProductListCMSPreview
Hipótesis principal: si el volumen residual procede de producción cliente, revisar primero PLP, PDP desde PLP, deeplinks y stock. Si procede de entornos internos o horarios de validación, revisar CMS preview/stage.

4. iOS: riesgo condicionado por flag

iOS parece estar migrado hacia StoreFront Menu para el caso nominal. La llamada funcional actual para menú es stores/{storeId}/menu, no /category. El riesgo está en el fallback legacy.

Flujo Request / acción Lectura funcional Referencias
Menú StoreFront GET stores/{storeId}/menu Carga el menú/categorías desde el nuevo modelo StoreFront. MenuEndpoint.swift, MenuManager.swift, CatalogViewController.swift
Items de menú GET stores/{storeId}/menu/items Endpoint disponible para resolver elementos concretos por itemId o seoId. No se observó como fuente principal de navegación normal. MenuItemsEndpoint.swift, MenuManager.swift
Fallback legacy categorías client?.categories Si enableStorefrontMenu está desactivada, puede volver a la petición legacy encapsulada en ITXClient. MenuManager.swift
Deeplinks y navegación Navegación sobre árbol ya cargado Se observan rutas de negocio de categoría, pero no una petición /category directa en ese punto. CategoryNavigationHandler.swift, CatalogViewController.swift
Validación clave: confirmar en configuración de últimas versiones que menu.enableStorefrontMenu está siempre activo para todos los mercados/entornos relevantes. Si está activo globalmente, iOS no debería ser el origen principal del residual.

5. Web: sin llamada literal, pero con acciones de categoría

En Web no se ha identificado construcción directa de una URL /category. La capa web utiliza acciones Corefront y wrappers; por tanto, el punto crítico es validar a qué URL final resuelve cada acción fuera del repo.

Flujo Acción / wrapper Lectura funcional Referencias
PLP / grid de categoría GET_PRODUCTS_FILTERED_PAGINATED_ORDERED Obtiene productos, filtros y ordenación para la parrilla de categoría. itxCategoryService.js
Hidratación de productos GET_CATALOG_PRODUCTS_ESI Convierte IDs de producto en tarjetas completas visibles en grid, búsqueda, PDP o módulos. itxCategoryService.js, productService.js, recommendationsService.js
Stock de categoría xRestGetCategoryStock / catalog:categoryStock Disponibilidad asociada a productos de categoría. Es el candidato web más cercano a una ruta category legacy. itx-request-api.js, itxCategoryService.js
Preview CMS / Stage Review GET_GRID, GET_PRODUCTS_FILTERED_PAGINATED_ORDERED, GET_CATALOG_PRODUCTS_ESI Validación interna de grids futuros con milestoneCode o gridId. stageReviewCategoryPage.js
Módulos de tienda/recomendación xRestGetCategory, producto filtrado y detalle de productos Algunos componentes usan una categoría como fuente para pintar módulos o CTAs. storeSectionsModal.js, recommendationsService.js
Validación clave: revisar el mapping Corefront de catalog:categoryStock, GET_PRODUCTS_FILTERED_PAGINATED_ORDERED y xRestGetCategory. El repo web no permite asegurar la URL final.

6. Falsos positivos descartados

  • Analítica: constantes como /category usadas para etiquetar pantallas o eventos no son llamadas HTTP de catálogo.
  • Navegación: rutas, deeplinks, URLs amigables y helpers de navegación pueden contener category, pero no necesariamente disparan la petición legacy.
  • Tests y mocks: fixtures como https://example.com/category no explican tráfico real.
  • Assets: rutas tipo /img/menu/categoryIcon son imágenes o recursos estáticos.
  • ProductsArray y product stock: usan categoryId como contexto, pero no son rutas /category.

7. Recomendaciones de siguiente paso

Prioridad Acción Objetivo
1 Confirmar en logs si el residual incluye plataforma, app version, endpoint exacto y user-agent. Separar Android real de tráfico interno, bots, preview o versiones antiguas.
2 En Android, revisar reachability de use cases legacy: GetWsCategoryListUC, GetWsCategoryUC, GetWsProductListUC, GetWsProductDetailUC, GetWsCategoryStockListUC. Determinar qué journeys siguen ejecutando rutas legacy en últimas versiones.
3 Validar flags/config de iOS: enableStorefrontMenu por mercado y entorno. Descartar fallback legacy de menú en producción.
4 Revisar mapping Corefront usado por Web para acciones de categoría y stock. Confirmar si alguna acción web resuelve internamente a la ruta migrada.
5 Cruzar timestamps de tráfico residual con eventos de preview, CMS, stage review o campañas. Identificar si el tráfico viene de operación interna y no de navegación cliente final.

8. Conclusión

La causa más probable de llamadas residuales a /category en últimas versiones es Android, porque mantiene endpoints explícitos y cadenas de ejecución conectadas con journeys de catálogo. iOS queda como riesgo de configuración por fallback legacy. Web no muestra llamada literal en el repo, pero requiere validación del mapeo de acciones Corefront para cerrar completamente la hipótesis.
